# Assessment Method

## Purpose

Define how architecture-led security assessments in this repository should be structured.

## Assessment Output Model

Each substantial assessment should produce:

1. an executive report
2. a detailed assessment or findings narrative
3. entries in the findings register
4. entries in the risk register
5. mapped remediation workstreams or ADRs

## Required Finding Fields

- finding ID
- title
- severity
- likelihood
- observation
- affected assets
- why it matters
- evidence
- risk statement
- recommendation
- related risk ID
- owner
- status

## Presentation Standard

- executive document should be concise and leadership-friendly
- detailed assessment should be evidence-driven and technically defensible
- registers should be normalized and easy to track over time
- diagrams and inventories should support reviewability without excess narrative
